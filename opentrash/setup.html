Phase 0 · One-time setup

Get your machine and your repository ready.

By the end of this guide you’ll have a working development environment, the opentrash repository created and cloned, an empty documentation site skeleton ready to grow, and the package name claimed. Works on macOS, Windows, and Linux. Plan about 90 minutes the first time.

Naming note: this guide uses opentrash as the package and repository name throughout. You can name yours anything — wherever you see opentrash, substitute your own name. Just be consistent.

Part 1 — Development environment

Five tools: Python (via miniforge), VS Code, git, the GitHub CLI, and UV. If you already have any of them, skip that step.

1 Install miniforge (Python 3.11)

Miniforge is a minimal Python distribution that uses conda-forge by default — lighter and cleaner than full Anaconda.

macOS (Apple Silicon)
curl -L -O "https://github.com/conda-forge/miniforge/releases/latest/download/Miniforge3-MacOSX-arm64.sh"
bash Miniforge3-MacOSX-arm64.sh

When prompted: accept the license (yes), accept the default install location, and answer yes to “initialize.” Then close and reopen Terminal.

macOS (Intel)
curl -L -O "https://github.com/conda-forge/miniforge/releases/latest/download/Miniforge3-MacOSX-x86_64.sh"
bash Miniforge3-MacOSX-x86_64.sh
Windows

Download Miniforge3-Windows-x86_64.exe from the miniforge releases page and run it. Choose Just Me, and on Advanced Options check “Add Miniforge3 to my PATH.” Afterwards use the Miniforge Prompt from the Start menu for all commands.

Linux
curl -L -O "https://github.com/conda-forge/miniforge/releases/latest/download/Miniforge3-Linux-x86_64.sh"
bash Miniforge3-Linux-x86_64.sh
Verify (all platforms)
conda --version

Shows conda 24.x.x or similar. If you see “command not found,” close the terminal completely and reopen it.

2 Install VS Code

Download from code.visualstudio.com and install. Then open the Extensions panel (left sidebar) and install:

  • Python (Microsoft)
  • Jupyter (Microsoft)
  • GitLens (GitKraken) — optional but helpful

macOS: make the code command available — in VS Code press Cmd+Shift+P, run “Shell Command: Install 'code' command in PATH.” On Windows/Linux the installer handles this.

code --version

3 Install git (and Homebrew on macOS)

macOS — Homebrew first, then git

Homebrew is the standard macOS package manager and makes the rest easy.

/bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)"

After install, follow the on-screen instructions to add Homebrew to your PATH (the installer prints the exact two lines for your shell). Then:

brew install git
Windows

Download from git-scm.com. During install, choose VS Code as the default editor if offered, and “Git from the command line and also from 3rd-party software” for PATH.

Linux
sudo apt install git
Configure your identity (all platforms)
git config --global user.name "Your Name"
git config --global user.email "you@example.com"
git --version

4 Install the GitHub CLI and sign in

The GitHub CLI (gh) is the fastest way to authenticate — no manual SSH keys needed. It handles credentials through a quick browser login.

macOS
brew install gh
Windows
winget install --id GitHub.cli

Or download the installer from cli.github.com.

Linux

Follow the install steps at cli/cli install guide.

Sign in (all platforms)
gh auth login

Choose GitHub.com → HTTPS → Yes (authenticate Git) → Login with a web browser. Copy the one-time code, press Enter, and approve in your browser. Done — far simpler than SSH keys.

Prefer SSH? If you’d rather use SSH keys (advanced), generate one with ssh-keygen -t ed25519 -C "you@example.com", add the public key at github.com/settings/keys, and verify with ssh -T git@github.com. The gh HTTPS path above is recommended for most people. 
gh auth status

5 Install UV

UV is a fast Python package manager we use for project dependencies.

macOS / Linux
curl -LsSf https://astral.sh/uv/install.sh | sh
Windows (PowerShell)
powershell -ExecutionPolicy ByPass -c "irm https://astral.sh/uv/install.ps1 | iex"

Restart your terminal, then verify:

uv --version

Part 2 — Create the repository

The opentrash package lives in its own repository (separate from any personal or org website repo). This keeps the package clean, independently versioned, and ready for PyPI — the standard layout for a real Python package.

6 Create and clone the repo

Create the repo on GitHub using the CLI (replace opentrash with your chosen name, and set the owner to your user or org):

gh repo create opentrash --public --description "Cloud-native geospatial tools for residential waste operations" --clone

Use --public for the eventual open release, or --private while you work. To create under an organization, use gh repo create YOUR-ORG/opentrash .... The --clone flag downloads it to your current folder.

Move into a sensible projects directory first if you like:

macOS / Linux
mkdir -p ~/Projects && cd ~/Projects
gh repo create opentrash --public --clone
Windows
mkdir %USERPROFILE%\Projects
cd %USERPROFILE%\Projects
gh repo create opentrash --public --clone

Then enter the repo and open it in VS Code:

cd opentrash
code .

Part 3 — The package skeleton + docs site

We create a minimal but complete skeleton now: the Python package folder, a documentation site (MkDocs with the Material theme), packaging config, and a workflow that auto-builds the docs to GitHub Pages. Every later lesson grows this.

7 Build the skeleton structure

This is the target layout. The starter files are provided with the course materials — drop them into your freshly cloned repo so it looks like this:

opentrash/ ├── opentrash/ # the Python package (code goes here) │ └── __init__.py # marks it a package; holds __version__ ├── docs/ # documentation site content │ └── index.md # docs homepage (placeholder for now) ├── .github/ │ └── workflows/ │ └── docs.yml # auto-build + deploy docs to GitHub Pages ├── mkdocs.yml # docs site config (Material theme) ├── pyproject.toml # packaging config; claims the name "opentrash" ├── README.md # repo front page ├── LICENSE # add your chosen license text └── .gitignore # Python ignores + project-specific

Where things go, in plain terms: your code lives in opentrash/. Your docs pages live in docs/. Configuration sits at the root. The .github/workflows/docs.yml file is what publishes the docs site automatically every time you push.

8 Create the environment and install tools

Create a dedicated conda environment for the project, then install MkDocs Material and the build tooling:

conda create -n opentrash python=3.11 -y
conda activate opentrash
pip install mkdocs-material build twine

mkdocs-material is the documentation theme, build creates the package distribution, and twine uploads it to PyPI (Part 4).

9 Preview the docs site locally

From the repo root, start the MkDocs live preview:

mkdocs serve

Open http://127.0.0.1:8000 in your browser. You’ll see the skeleton docs homepage. Edits to anything in docs/ reload live. Press Ctrl+C to stop.

10 Enable GitHub Pages for the docs

The included docs.yml workflow builds the site and pushes it to a branch called gh-pages on every commit to main. After your first push (Step 12), enable Pages:

  1. On GitHub, open the repo → Settings → Pages.
  2. Under Build and deployment, set Source to “Deploy from a branch.”
  3. Set the branch to gh-pages and folder to / (root). Save.
Default URL (no custom domain)

Your docs will be live at https://<owner>.github.io/opentrash/. Nothing else to do.

Optional: custom domain

If you own a domain (for example opentrash.app), you can point it at this repo’s Pages site. Each repository supports one custom domain, so a dedicated package repo can use its own domain independently of any other site you host.

  1. In Settings → Pages → Custom domain, enter your domain and Save. This writes a CNAME file to the repo.
  2. At your DNS provider, add the records GitHub specifies — typically a CNAME record pointing your domain to <owner>.github.io (and/or A records for an apex domain). GitHub’s Pages settings page shows exactly which records to add.
  3. Wait for DNS to propagate, then enable “Enforce HTTPS.”

You can switch between the default URL and a custom domain at any time — this is a setting, not a one-way door. Start with the default if you’re unsure.

Part 4 — Claim the package name on PyPI

Package names on PyPI are first-come, first-served, and claimed by uploading a release. Publishing a tiny 0.0.0 placeholder now reserves your name while it’s available. (You’ll learn the full publishing workflow properly in Lesson 12 — this is just to hold the name.)

11 Build and upload a placeholder release

First, create a free account at pypi.org and check your name is available at https://pypi.org/project/opentrash/.

With pyproject.toml set to name = "opentrash" and version = "0.0.0", from the repo root:

python -m build
twine upload dist/*

twine will prompt for your PyPI credentials (use an API token from your PyPI account settings). Once uploaded, the name is yours. Verify at https://pypi.org/project/opentrash/.

Naming reminder: if you chose a different package name, make sure pyproject.toml, the repo, and the folder all match it. Consistency now saves confusion later.

Part 5 — Commit and verify

12 First commit and push

git add .
git commit -m "Phase 0: package skeleton, docs scaffold, packaging config"
git push origin main

If your default branch is master, use that instead of main.

Final checklist

  • conda --version, code --version, git --version, gh --version, uv --version all work
  • gh auth status shows you signed in
  • Repo created and cloned locally
  • Skeleton structure in place (package folder, docs, mkdocs.yml, pyproject.toml, workflow)
  • mkdocs serve renders the docs homepage locally
  • First push succeeded; GitHub Pages enabled (default URL or custom domain)
  • Package name claimed on PyPI
When every box is checked, Phase 0 is complete. You have a working machine, a live (or ready-to-go-live) docs site, and your package name secured. Lesson 1 picks up from here — turning this skeleton into a real package.

Troubleshooting

  • conda: command not found — the terminal didn’t reload after install. Close it completely and open a fresh one.
  • gh: command not found — reopen the terminal; on macOS confirm brew finished and is on your PATH.
  • Auth fails in browser — rerun gh auth login and make sure you copy the full one-time code.
  • Pages shows 404 — check Settings → Pages: source should be the gh-pages branch, root folder. The first build can take a couple of minutes.
  • twine rejects upload — the name may already be taken, or your version already exists. Bump the version or pick another name.
  • Anything else — capture the exact error text and ask. Don’t suffer silently; a clear question is half the fix.
