Lesson 7 of 12 · Refined Hits

From raw GPS to parcel-level service events — cheaply.

A refined hit is one row that says: vehicle V was at parcel P at time T, within edge distance D feet, going S mph. This is the atom of every downstream report and the input to pattern detection. Building it from raw GPS over years of history is the kind of work that brings naive approaches to their knees; we use the indexes from Lesson 6 to make it tractable.

Time: ~75 min You'll touch: patterns/refined_builder Result: the refined-hits cache that pattern detection reads

· Objective

Build a single module — patterns/refined_builder.py — with two complementary paths, plus a codified schema constant. The lesson frames the two modes around how the package will actually be used.

  • Mode A — merge_existing (primary). Read existing refined-hits parquets, filter by date window, write a single output. Used every time you re-run pattern analysis with different knobs — cheap and fast.
  • Mode B — raw_match (bootstrap). Build refined hits from raw GPS + parcels + the master index. Used once when standing up a new agency or iterating on the refinement logic.
  • REFINED_HITS_COLUMNS — the canonical column order. Downstream code depends on it.
Two paths, one philosophy. In production, the streaming engine produces refined hits incrementally as pings arrive (the Postgres adapter from Lesson 6 feeds it). Downstream tools always read via Mode A. Mode B is the one-time bootstrap that gets a new install up to speed, or the dev tool when you’re iterating on the refinement logic.

· Before you begin

  • You’ve completed Lesson 6 — the master GPS index and parcels-WKB are built.
  • No new dependencies. Shapely and pyproj are already core deps from Lesson 2.
  • To run Mode B against real data you need: raw GPS cache, parcels-with-WKB parquet, and the master GPS index. The tests use synthetic 3-parcel / 4-ping fixtures, so you can build and verify the module without any of that.

· Build it, step by step

1 Codify the schema and knobs

Two constants at the top of the module:

  • REFINED_HITS_COLUMNS — the canonical 11 columns in order. Downstream readers depend on this shape, so it’s a stable contract, not a comment.
  • RefinedBuilderConfig — a frozen dataclass bundling the knobs (speed_min, speed_max, edge_dist_ft, batch_size, flush_every). Callers pass one object instead of ten arguments.
REFINED_HITS_COLUMNS = (
    "APN", "route_ref", "route_org", "route_rec",
    "VehicleName", "DeviceId",
    "dt_local", "speed", "lat", "lon", "dist_edge_ft",
)

@dataclass(frozen=True)
class RefinedBuilderConfig:
    speed_min: float = 0.0    # parcels are visited slowly
    speed_max: float = 5.0
    edge_dist_ft: float = 50.0
    batch_size: int = 30      # raw GPS files per DuckDB query
    flush_every: int = 100    # batches per chunk dump

2 Mode A — merge_existing_refined()

The primary path. Read one or more refined-hits parquets via DuckDB’s read_parquet([...], union_by_name=true) (so source files don’t have to be column-identical), filter to a date window if requested, SELECT DISTINCT to dedupe, write ZSTD-compressed output:

merge_existing_refined(
    sources=[refined_2024.parquet, refined_2025.parquet],
    out_path="refined_2024_2025.parquet",
    date_start="2024-01-01", date_end="2025-12-31",
)

This is the function pattern detection (Lesson 8) calls every time it runs. Re-runs are cheap because there’s no spatial join — just a columnar scan of parquet.

3 Mode B — candidate match (DuckDB)

The bootstrap path’s first half. For a batch of raw GPS files:

  1. Look up which route_ids each file belongs to (from the master index slice).
  2. Pull the parcels assigned to those routes (from parcels-with-WKB).
  3. Join GPS × parcels with a bbox prefilter (route-aware, so we only check parcels on the route the file’s vehicle drove) and a speed window ([0, 5] mph — trucks at parcels are slow).
JOIN batch_parcels p
  ON p.route_ref = bp.route_id
 AND g.Longitude BETWEEN p.min_lon - deg_lon AND p.max_lon + deg_lon
 AND g.Latitude  BETWEEN p.min_lat - deg_lat AND p.max_lat + deg_lat
WHERE g.Speed BETWEEN 0 AND 5

The deg_lat/deg_lon padding converts 50 ft into lat/lon-degree units (conservatively, so we over-include rather than miss). Cheap, vectorized, runs entirely in DuckDB.

4 Mode B — edge refinement (Shapely + pyproj)

The bootstrap path’s second half. For each chunk of candidates from step 3:

  1. Haversine pre-filter — coarse great-circle distance from the ping to the parcel centroid, vectorized in NumPy. Drop anything > 5× edge_dist_ft.
  2. Exact edge distance — for survivors, use Shapely’s nearest_points() on the parcel polygon and pyproj.Geod.inv() for the great-circle distance to the nearest edge, converted to feet.
  3. Keep rows where dist_edge_ft ≤ 50.

Polygons are loaded lazily from WKB and cached by APN, so we never round-trip Shapely for a parcel more than once per chunk.

5 Mode B — batch + safe dump

The reason a 3-year, 250-vehicle rebuild fits in memory: we never hold more than flush_every batches at once. Every flush_every batches, candidates are concatenated and written to chunk_NNNN.parquet, then refined to refined_NNNN.parquet, then the in-memory list is cleared. At the end, all chunk files are merged via Mode A:

refined_paths = raw_match_to_chunks(
    gps_files, route_ptr, parcel_bbox_ref, apn_wkb,
    chunk_dir="chunks/",
    config=RefinedBuilderConfig(batch_size=30, flush_every=100),
)
merge_refined_chunks(refined_paths, out_path="refined_full.parquet")

Idempotent re-runs: chunks live on disk, so an interrupted run picks up cleanly. This is the "batch-safe dumping" pattern that’s the central theme of Lesson 8 too — never let one giant pandas frame live in memory.

6 Test without the long run

Cover the schema, the config defaults, Mode A merge + date window + dedupe, the haversine + edge refinement on a 2-ping fixture (one inside parcel, one outside), and an end-to-end Mode B with 3 synthetic parcels + 4 pings (slow-near, slow-near, fast-near drops, far-away drops). No 18-hour batch, no real data:

pytest tests/test_refined_builder.py -v
git add . && git commit -m "Lesson 7: refined hits (Mode A primary, Mode B bootstrap)"
git push origin main

· What you built

  • A codified refined-hits schema (REFINED_HITS_COLUMNS) that downstream code depends on.
  • Mode A — merge existing refined parquets with date filtering and dedup. The path pattern detection calls every run.
  • Mode B — candidate match via bbox prefilter, then exact-edge refinement with Shapely + pyproj.
  • A batch + safe-dump loop that keeps memory flat across multi-year rebuilds.
  • Tests covering both modes with synthetic data — no live API, no real geometries needed.
You now have the refined-hits artifact every downstream tool reads. Next lesson takes refined hits and detects service patterns: which parcels are served weekly, which biweekly, which Tuesday at 9 AM by truck 815001.

· Companion resources

Optional, for going deeper.

  • Shapely: nearest_points — the operation behind edge refinement.
  • pyproj.Geod: geodesic distance — exact great-circle distance on the WGS84 ellipsoid.
  • Haversine formula: the cheap great-circle approximation we use for the pre-filter — fast enough for millions of rows per second in NumPy.
