Lesson 2 of 12 · The Foundation

Coordinate systems, parcels, and the compute engine.

In Lesson 1 you made the package install and run. Now you’ll write its first real code — the spatial bedrock everything else stands on: how the package handles coordinate reference systems, how it loads the parcel layer, and how it talks to DuckDB. Small, foundational, and the thing every later lesson builds on top of.

Time: ~60 min You'll touch: core/crs · core/duckdb_session · prep/parcels Result: load parcels, reproject, query DuckDB

· Objective

By the end of this lesson, the package can load real spatial data and reason about it correctly. Specifically, you’ll build:

  • core/crs.py — the reprojection convention: work in a projected CRS (EPSG:2230, feet), output in WGS84 (EPSG:4326) for web maps.
  • core/duckdb_session.py — a configured DuckDB connection with the spatial extension loaded.
  • prep/parcels.py — load the parcel layer, enforce CRS, and the bounding-box prefilter that keeps spatial work fast.
  • Real tests for each, using tiny synthetic geometries (no operational data needed).
One convention runs through this whole lesson: store and compute in a projected CRS, convert to lon/lat only at the rendering boundary. Get this right once and every distance, area, and spatial join downstream is correct. Get it wrong and the bugs are subtle and everywhere.

· Before you begin

  • You’ve completed Lesson 1 — the package installs, runs, and tests pass.
  • Your environment is active: conda activate opentrash.
  • You have a parcel file to experiment with. Many counties publish parcels openly; a GeoParquet or GeoPackage subset for your area is ideal. The code also works against synthetic data, which is all the tests need.

· Build it, step by step

1 Add the spatial dependencies

The package needs real libraries now. Update the dependencies list in pyproject.toml:

dependencies = [
    "geopandas>=0.14",
    "shapely>=2.0",
    "pyproj>=3.6",
    "duckdb>=1.0",
]

Then reinstall so the new deps are present:

pip install -e ".[dev]"

GeoPandas pulls in Shapely and PyProj transitively, but listing them explicitly documents intent — we use each directly.

2 Write core/crs.py

Create the CRS module. Two named constants and two helpers:

  • WORKING_CRS = "EPSG:2230" — the projected CRS we compute in (California State Plane Zone 6, US survey feet). Set this to your own data’s CRS if you’re elsewhere.
  • WEB_CRS = "EPSG:4326" — WGS84 lon/lat, for web-map output.
  • to_working(gdf) and to_web(gdf) — reproject helpers that refuse to guess when a GeoDataFrame has no CRS (they raise instead).

Refusing to guess on a missing CRS is deliberate. A silent assumption here produces coordinates that look plausible but are wrong — the worst kind of bug. Fail loud, fail early.

3 Write core/duckdb_session.py

A single connect() function that opens a DuckDB connection and loads the spatial extension (installing it on first use if needed). This is the serverless compute engine — it reads parquet straight off disk, no server.

con = connect()              # in-memory by default
con.execute("SELECT ST_AsText(ST_Point(1, 2));")

Try LOAD first and only INSTALL if that fails — it avoids a network round-trip on machines where the extension is already present.

4 Write prep/parcels.py

Three functions that make the parcel layer usable:

  • load_parcels(path) — read a parcel file (GeoParquet/GeoPackage), enforce a CRS, reproject to the working CRS.
  • bbox_filter(gdf, minx, miny, maxx, maxy) — the cheap first pass: a spatial-index-backed bounding-box query, far faster than exact intersection over the whole layer.
  • bbox_from_geometry(geom, buffer=0) — derive a (padded) bounding box from a route or other geometry.

The bbox prefilter is the performance heart of the package: narrow by cheap bounding box first, then do expensive exact geometry work on the small result. This is how it stays fast on a laptop over hundreds of thousands of parcels.

5 Write the tests

Add tests/test_crs.py and tests/test_parcels.py. Build tiny synthetic geometries with Shapely (a few box(...) squares), write them to a temp GeoPackage where a file is needed, and assert:

  • web output lands in plausible lon/lat for your region;
  • a CRS-less frame raises;
  • bbox_filter narrows to the expected parcels;
  • buffering expands the box symmetrically.

Synthetic geometries keep the tests fast, deterministic, and free of operational data — they run anywhere, including CI.

6 Run everything, then commit

ruff check .
pytest -v

All tests should pass. Then commit and push — CI runs the same checks on GitHub:

git add .
git commit -m "Lesson 2: CRS handling, DuckDB session, parcel layer"
git push origin main

· What you built

  • A CRS convention the whole package follows — compute projected, output lon/lat.
  • A DuckDB session with spatial SQL ready to go.
  • A parcel loader with the bounding-box prefilter that keeps everything fast.
  • Real tests over synthetic geometry, green locally and in CI.
This is the bedrock. From here, every lesson adds a layer that assumes correct coordinates, fast spatial narrowing, and a DuckDB engine on hand. Lesson 3 starts using them on real operational data — the sites layer.
