Lesson 5 of 12 · Packaging Basics

Notebook code → library code: dataclasses, types, and Protocols.

A notebook is a great place to figure something out. It’s a terrible place to keep something that others depend on. In this lesson you’ll add two small pieces of new code that demonstrate the patterns every later lesson leans on: a dataclass for structured values, type hints everywhere, and a Protocol that lets the package swap implementations without changing the consumers.

Time: ~60 min You'll touch: core/vehicle_ids · adapters/gps/base Result: the packaging idioms in place

· Objective

Build two small, real-code pieces that introduce the patterns the rest of the package follows. You’re not refactoring — you’re adding new files that will be used immediately in Lesson 6.

  • core/vehicle_ids.py — a VehicleId dataclass + classification table as data + an is_our_fleet() filter.
  • adapters/gps/base.py — a GPSAdapter Protocol defining the canonical GPS data shape every vendor implementation must satisfy.
  • Tests for both: parse / classify / filter behavior, and structural-typing assertions for the Protocol.
The theme is vendor-independence and editable knobs. A real package can’t hardcode one GPS provider, and operational rules (which fleet prefixes mean what) belong in a table you can edit — not in scattered if statements. These two files demonstrate both.

  • You’ve completed Lesson 4 — tonnage ingest works.
  • Environment active.
  • No new dependencies. Everything in this lesson uses the standard library plus pandas (which is already installed).

· Build it, step by step

1 Why these patterns, briefly

Three pieces of the standard library carry a lot of weight in a real package:

  • Dataclasses — structured values with fields, equality, immutability, and a clean repr() for free. Replaces “a dict with the right keys”.
  • Type hints — x: int, -> str, etc. Free documentation; your editor catches bugs before they run; ruff and mypy enforce them. Cost: minimal.
  • Protocols (typing.Protocol) — an interface defined by shape, not inheritance. Any class with the right methods satisfies it. The right way to plan for “swap this vendor for another” without writing an abstract base class hierarchy.

2 Write core/vehicle_ids.py

The vehicle ID is a short numeric code: the first three digits are a fleet prefix identifying the truck’s kind (automated, manual, etc.); the rest is the per-truck number. Build it:

  • Classification table as data — a plain FLEET_CLASSIFICATION dict mapping prefix to kind. Edit one table when ops changes; the rest of the package picks it up.
  • VehicleId dataclass with frozen=True, slots=True: immutable, hashable, memory-tight. Three fields: raw, prefix, number; a kind property derived from the table.
  • parse_vehicle_id(value) — strip Excel noise (commas, trailing .0, whitespace), reject NA-token strings, return a VehicleId or None. Returning None rather than raising is deliberate — callers process columns of mixed-quality data and want to skip misses, not abort.
  • is_our_fleet(value, prefixes=OUR_FLEET_PREFIXES) — the filter the tonnage cleaner needs. Pass your own tuple to widen or narrow.

This module is “the file to edit first” when adapting the package to another agency. Make that obvious in the module docstring — future you, or someone forking, will appreciate it.

3 Write adapters/gps/base.py

Today GPS comes from Geotab. Next year it might be Samsara, or a CSV dump from a partner. The package shouldn’t care. Define the shape every adapter must satisfy:

@runtime_checkable
class GPSAdapter(Protocol):
    def fetch(
        self,
        vehicle: str,
        start_date: date | str,
        end_date: date | str,
    ) -> pd.DataFrame: ...

Plus a canonical output schema the Protocol promises: a tuple of (column, dtype) pairs documenting exactly what every adapter returns.

GPS_SCHEMA = (
    ("vehicle_id", "string"),
    ("dt_utc",     "datetime64[ns, UTC]"),
    ("dt_local",   "datetime64[ns, US/Pacific]"),
    ("lat",        "float64"),
    ("lon",        "float64"),
    ("speed_mph",  "float64"),
)

Note @runtime_checkable: it enables isinstance(x, GPSAdapter) as a sanity check. Adapters don’t subclass GPSAdapter — structural typing means any class with a matching fetch is a GPSAdapter.

4 Tests — including a structural-typing one

For vehicle_ids: confirm parse handles all the Excel noise variants, the dataclass is immutable + hashable, classification looks up the right kind, and is_our_fleet filters correctly with both default and custom prefixes.

For adapters/gps/base: write a tiny fake adapter class with a matching fetch and assert isinstance(fake, GPSAdapter) is True. Write another class without one and assert it’s False. That’s the Protocol working.

Testing a Protocol is mostly testing the constants and the structural check. The real test of a Protocol comes when the first implementation lands — in this case, the Geotab adapter in Lesson 6.
