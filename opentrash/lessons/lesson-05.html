Lesson 5 of 12 · GPS & Caching

Pull GPS traces — behind a Protocol, and cached so you pull once.

Now the package reaches out to the world. Every collection truck carries a GPS unit; we pull its trace for a given day from the vendor’s API. But we do it the right way: behind a vendor-neutral adapter (so Geotab can be swapped for anything later) and a cache-first layer (so a day already pulled is read from disk, never re-fetched). You’ll put the VehicleId from Lesson 4 to work here.

Time: ~75 min You'll touch: adapters/gps/{base,geotab} · routeview/gps_cache Result: cached GPS pulls, one file per vehicle per day

· Objective

Build the GPS subsystem in three layers: a Protocol that defines the shape, a concrete Geotab adapter that satisfies it, and a cache that makes repeated pulls free.

  • adapters/gps/base.py — the GPSAdapter Protocol + the canonical output schema every adapter returns.
  • adapters/gps/geotab.py — GeotabAdapter: pulls LogRecords, converts local-day ↔ UTC, normalizes to the canonical schema. Credentials are arguments, never hardcoded.
  • routeview/gps_cache.py — cache-first retrieval, stored as cache/YYYYMMDD/<vehicle>.parquet.
Two patterns carry this lesson: the adapter Protocol (decouple the package from any one GPS vendor) and cache-first I/O (never pay for the same slow, rate-limited pull twice). Both are everywhere in production data work.

· Before you begin

  • You’ve completed Lesson 4 — you have VehicleId and the fleet filter.
  • Install the optional GPS extras: pip install -e ".[geotab]" (pulls in mygeotab and pytz).
  • Geotab credentials, supplied as arguments or environment variables — never committed to the repo. If you ever pasted credentials into a script, rotate them.

· Build it, step by step

1 Define the Protocol — adapters/gps/base.py

Start with the interface, not the implementation. A GPSAdapter Protocol with one method, fetch(vehicle, start_date, end_date) -> DataFrame, plus a canonical schema the Protocol promises every adapter returns:

GPS_SCHEMA = (
    ("vehicle_id", "string"),
    ("dt_utc",     "datetime64[ns, UTC]"),
    ("dt_local",   "datetime64[ns, US/Pacific]"),
    ("lat",        "float64"),
    ("lon",        "float64"),
    ("speed_mph",  "float64"),
)

Defining the shape first means the cache and everything downstream can be written and tested against the Protocol, before any real vendor code exists.

2 Implement adapters/gps/geotab.py

A GeotabAdapter dataclass that satisfies the Protocol. Faithful to the operational pull, with the rough edges handled:

  • Credentials as arguments (username, password, database, server), falling back to GEOTAB_* environment variables. Validated before any network or vendor import — a clear error if missing.
  • Local-day ↔ UTC conversion. The caller passes local (PT) calendar days; Geotab wants a UTC instant window. Convert the local-day bounds to UTC for the query, then filter results back to the local day (a UTC window spans two local dates at the edges).
  • Lazy import of mygeotab inside fetch — the package doesn’t hard-depend on a vendor SDK.
  • Normalize to the canonical schema: speed km/h → mph, datetimes to UTC + local, the 6 columns, sorted by time. Returns the DataFrame — it does not write a file (that’s the cache’s job).

Splitting the row-normalization into a static helper (_normalize_rows) means you can unit-test the parsing/filtering logic without ever touching the live API.

3 Build the cache — routeview/gps_cache.py

GPS pulls are slow, rate-limited, and historical data never changes — the perfect case for a cache. Lay it out by day, one file per vehicle:

cache/
  20250818/
    815421.parquet
    832111.parquet
  20250819/
    815421.parquet

Two functions:

  • get_gps_day(adapter, vehicle, day, cache_dir) — cache hit → read parquet; cache miss → adapter.fetch(...), write, return. A refresh=True flag forces a re-pull.
  • get_gps_fleet_day(adapter, vehicles, day, cache_dir) — loop a fleet list, cache each independently, concatenate. A partially-cached fleet only fetches the missing trucks.

The cache takes any GPSAdapter — it never mentions Geotab. Swap vendors and the cache code doesn’t change. That’s the Protocol paying off.

4 Test without the network

The live API can’t be in the test suite. Test around it:

  • Geotab: the pure helpers (date coercion, km/h→mph), credential validation (missing creds raise), and _normalize_rows (raw pings → canonical schema, local-day filter) — all without a real call. Plus isinstance(adapter, GPSAdapter).
  • Cache: a stub adapter that returns a canned frame and counts calls. Prove a miss fetches + writes, a hit reads with no second call, refresh re-fetches, and the on-disk layout is YYYYMMDD/<vehicle>.parquet.

· What you built

  • A vendor-neutral GPS Protocol — the package depends on a shape, not a vendor.
  • A Geotab adapter with safe credential handling and correct local/UTC time logic.
  • A cache-first layer storing one parquet per vehicle per day — pulls become free on repeat.
  • Tests that fully exercise the logic without touching the live API.
You can now assemble a day’s GPS for a whole fleet, fast, from cache. The next stretch of the course turns these traces into insight: where trucks stopped, which parcels they served, and the weekly patterns hidden in the pings.
