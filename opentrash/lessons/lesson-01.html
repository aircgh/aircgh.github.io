Lesson 1 of 12 · Orientation

From empty skeleton to a real, installable package.

You set up your machine and created the repository in the Setup Guide. Now you’ll give the package its skeleton anatomy, make it installable in editable mode, and wire up a working command-line interface — the foundation every later lesson builds on.

Time: ~45 min You'll touch: pyproject.toml · CLI · tests Result: opentrash --help works

· Objective

By the end of this lesson, your package will install, run, and test green. Specifically:

  • pip install -e . installs the package in editable mode.
  • opentrash --version and opentrash --help work.
  • The full module anatomy exists (core, prep, tonnage, patterns, routeview, adapters), each documented.
  • pytest runs and passes a first smoke test.
We’re building anatomy, not features. The analytical code comes in later lessons. Lesson 1 is about a package that stands up cleanly and proves every piece of the toolchain — install, run, test — works end to end.

· Before you begin

  • You’ve completed the Setup Guide — tools installed, repo created and cloned, name claimed on PyPI.
  • Your conda environment is active: conda activate opentrash.
  • You’re in the repo folder in your terminal and have it open in VS Code (code .).

· Build it, step by step

1 Give the package its module anatomy

Inside the opentrash/ package folder, create one folder per subsystem, each with an __init__.py. The target layout:

opentrash/ ├── __init__.py ├── cli.py ├── core/ # shared primitives (CRS, IO, DuckDB, hashing) │ └── __init__.py ├── prep/ # one-time data prep (parcels, sites, layers) │ └── __init__.py ├── tonnage/ # ingest + normalize tonnage │ └── __init__.py ├── patterns/ # per-parcel service signatures │ └── __init__.py ├── routeview/ # build the RouteView HTML │ └── __init__.py └── adapters/ # pluggable data sources ├── __init__.py └── gps/ └── __init__.py

Each __init__.py holds a docstring describing what will live there. Writing the intent now keeps the package self-documenting as it grows — and helps you (and anyone reading) remember the plan.

2 Write the CLI

Create opentrash/cli.py using Python’s standard-library argparse — no extra dependency. It exposes --version, --help, and four subcommands (routeview, patterns, prep, tonnage) that are stubs for now: each prints a friendly “not implemented yet” and exits cleanly.

Stubbing the subcommands shows the package’s shape without pretending it does things it can’t. Later lessons replace each stub with real logic.

3 Upgrade pyproject.toml

Two changes from the Phase 0 skeleton:

  • Bump the version to 0.1.0.dev0 — signalling “development started, not yet released” (the 0.0.0 was just the name claim).
  • Add a [project.scripts] entry so installing the package creates the opentrash command:
[project.scripts]
opentrash = "opentrash.cli:main"

This line is what makes opentrash --help work from your shell after install — it points the command at main() in cli.py.

4 Add a first test

Create tests/test_smoke.py with three trivial checks: the package exposes a version, the CLI runs with no arguments, and a stub subcommand returns cleanly. Add a [tool.pytest.ini_options] block to pyproject.toml pointing testpaths at tests.

These “smoke tests” aren’t about logic — they prove the scaffolding holds together. As real features land, real tests join them.

5 Install, run, and test

From the repo root, with your environment active:

pip install -e ".[dev]"
opentrash --version
opentrash --help
pytest -v

You should see the version print, the help list all four subcommands, and three tests pass. If opentrash isn’t found, confirm the install succeeded and your environment is active.

6 Add continuous integration

Create .github/workflows/tests.yml — a workflow that installs the package, runs ruff (lint), and runs pytest on every push and pull request. Now the repo checks itself every time you push.

7 Document the architecture & commit

Add a docs/architecture.md page describing the module map and data flow, and register it in mkdocs.yml’s nav. Preview with mkdocs serve, then commit and push:

git add .
git commit -m "Lesson 1: package anatomy, CLI, smoke tests, CI"
git push origin main

· What you built

  • A package that installs in editable mode and exposes a real opentrash command.
  • The full module skeleton — every subsystem has a home and a documented purpose.
  • A working CLI with stubbed subcommands that show the shape of what’s coming.
  • A passing test suite and CI that runs it automatically.
  • An architecture docs page on your growing documentation site.
Your package is now a real, installable, self-checking project — just without the analytical features yet. That’s exactly where Lesson 2 picks up.
