Lesson 6 of 12 · Indexing & Postgres

Small indexes to find big data — the central theme.

You can’t spatial-join hundreds of millions of GPS pings against a quarter million parcels by brute force — not on a laptop, not in a cloud notebook, not anywhere. The trick is indexing: a few small summary tables that let you find which big files to open in milliseconds. This lesson builds those indexes, and adds the Postgres adapter that’s the production GPS source.

Time: ~90 min You'll touch: adapters/gps/postgres · prep/{static_layers,parcels_wkb} · routeview/{gps_indexes,master_index} Result: “files for route X, window Y” in milliseconds

· Objective

Build four small indexes and one streaming adapter. Together they turn an intractable spatial-join into a series of fast lookups, and connect the package to the production data source.

  • adapters/gps/postgres.py — a second GPSAdapter for the streaming Postgres database (production data path).
  • prep/static_layers.py — route polygons (AUTO + HTC) → buffered 4326 bboxes.
  • prep/parcels_wkb.py — parcels with WKB geometry + per-route assignment + bbox columns.
  • routeview/gps_indexes.py — per-day vehicle-day summaries (one row per vehicle per day).
  • routeview/master_index.py — the STAC-like lookup that joins them: vehicle-day bbox ↔ route bbox.
The Lesson 5 Geotab adapter and this lesson’s Postgres adapter implement the same Protocol. They land pings in the same cache layout, and downstream code consumes them identically. That’s the Protocol paying off — swapping vendors changes one line.

· Before you begin

  • You’ve completed Lesson 5 — the GPS Protocol and cache are in place.
  • Install the Postgres extra: pip install -e ".[postgres]" (sqlalchemy + psycopg2). Skip it if you only have Geotab; the rest of the lesson runs without it.
  • For the indexing pieces you need some parcels and route polygons. The tests use tiny synthetic geometries, so you can build and verify everything before bringing in real data.
  • Heads up — cache directory format: Lesson 5 used YYYYMMDD; Lesson 6 standardizes on YYYY-MM-DD to match real ops directories and what the Postgres adapter writes. The Lesson 5 module is updated to match.

· Build it, step by step

1 Add the Postgres adapter — adapters/gps/postgres.py

Production GPS comes from a Postgres database where a streaming adapter writes pings every ~10 seconds. A PostgresAdapter dataclass implementing GPSAdapter:

  • Connection URL passed as an argument (or OPENTRASH_PG_URL environment variable) — never hardcoded.
  • Local-day window → UTC half-open range; queries LogRecords2 joined to Devices2 for vehicle names.
  • Server-side chunked fetch (stream_results=True, fetchmany(chunksize)) so memory stays flat on big windows.
  • Normalizes to the canonical 6-column schema (speed km/h → mph, datetimes to UTC + Pacific).

Override devices_sql / pings_sql on the constructor if your schema differs. The default queries match the Geotab streaming integration shape.

2 Route polygons → bboxes — prep/static_layers.py

Routes come in two flavors in this operation:

  • AUTO routes — large engulfing polygons (~1,200+ parcels each). A ping anywhere on a served street falls inside the polygon. No buffer needed.
  • HTC routes (Hard-to-Collect / manual) — the polygons are parcel-shape copies, not engulfing. A ping in the street next to a parcel can fall outside. We compensate by buffering HTC by 150 ft in the working CRS (default knob) before extracting bboxes.

build_route_index(auto_path, htc_path, htc_buffer_ft=150) handles both: AUTO takes precedence (any route in both layers uses the AUTO geometry), HTC-only routes get the buffer, all reduced to one DataFrame of (route_id, rte_min_lon, rte_min_lat, rte_max_lon, rte_max_lat) in EPSG:4326.

3 Parcels with WKB — prep/parcels_wkb.py

The refined-hits builder (next lesson) needs three things from each parcel: a fast bbox prefilter, the exact polygon for nearest-point distance, and the route assignments. Pack them into a single parquet:

APN  UNITQTY  route_ref  route_org  route_rec
min_lon  min_lat  max_lon  max_lat
geom_wkb     # the polygon as WKB bytes

WKB lets downstream tools open the file with plain read_parquet (no geo dependencies) and only pay the Shapely cost for the parcels they actually need.

4 Per-day vehicle-day indexes — routeview/gps_indexes.py

For every day in the cache, summarize each vehicle’s parquet into one row: bounding box, time range, row count, file path. The summary is tiny — a year of pings becomes ~365 small index files of ~hundreds of rows each.

cache/
  2026-01-18/
    815421.parquet            # raw pings, big
    832111.parquet
  indexes/
    2026-01-18/
      vehicle_day_index.parquet   # one row per vehicle, small

build_all_day_indexes(cache_dir) is idempotent — existing day-indexes are skipped unless you pass overwrite=True. Run it after each Postgres pull and the indexes always reflect what’s on disk.

5 The master index — routeview/master_index.py

This is the payoff. Bbox-overlap join between vehicle-day summaries and route bboxes, computed in DuckDB:

SELECT v.*, r.route_id, r.rte_*
FROM vehicle_day_indexes v
JOIN route_bboxes r
  ON NOT (v.lon_max < r.rte_min_lon OR v.lon_min > r.rte_max_lon
       OR v.lat_max < r.rte_min_lat OR v.lat_min > r.rte_max_lat)

Result: one row per (day, vehicle, route) triple where bboxes intersect — typically tens of millions of rows but well under 200 MB on disk. Now "which GPS files touch route 22064 in Jan 2026" is a DataFrame filter that returns in milliseconds.

A vehicle can match many routes (it drove through several); a route matches many vehicle-days (the fleet covers it over time). One bbox compare, millions of pings effectively narrowed without ever opening them.

6 Test everything, then commit

Tests use synthetic geometries and a fake DB result — no live Postgres needed. Verify the Postgres adapter’s pure helpers and Protocol conformance, the bbox math (dissolve-by-route, buffer in feet, edge-buffer in degrees), the parcels-WKB round-trip, the day-index summarizer’s idempotency, and the master-index join correctness.

pip install -e ".[dev,postgres]"
ruff check .
pytest -v
git add .
git commit -m "Lesson 6: indexing infrastructure + Postgres adapter"
git push origin main

· What you built

  • A Postgres adapter that satisfies the same Protocol as Geotab — production GPS streams in cleanly.
  • Route bboxes with the AUTO vs HTC buffer convention captured as a documented knob.
  • Parcels with WKB — one parquet that downstream tools open quickly and only Shapely-parse when needed.
  • Per-day vehicle-day summaries that turn a cache full of big files into a queryable index.
  • A master GPS index that makes “which files for route X, window Y” instant.
  • Tests covering all of it, without a live database or real geometries.
The substrate is now in place. Next lesson uses every one of these indexes to build refined hits — the GPS pings that actually landed on parcels — without ever doing a brute-force spatial join over the full dataset.
