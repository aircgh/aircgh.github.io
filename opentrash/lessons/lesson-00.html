Lesson 0 of 12 · Optional on-ramp

From notebook to package: you already know how, you just follow some rules now.

Most of this course builds a real Python package, step by step. But if you live in Jupyter notebooks and haven’t done much “software engineering,” the jump can feel intimidating. It shouldn’t. This short lesson shows that the gap between “code that works in a notebook” and “code that ships as a package” is mostly a handful of conventions — not a different skill.

This lesson is optional. If you’re already comfortable with functions, modules, classes, and packaging, skip straight to Lesson 1. If not, spend 30 minutes here first — the rest of the course will feel much smoother. This is an on-ramp, not a deep dive; pointers to go deeper are at the end.

· What actually changes

A notebook is brilliant for figuring things out: run a cell, look at the output, tweak, repeat. But notebooks are a poor place to keep code others depend on — cells run out of order, state lingers, and there’s no clean way to import “that function from cell 12” into another project.

Turning notebook code into package code mostly means applying a few rules:

  • Put related functions in a module (a .py file) so they can be imported anywhere.
  • Give functions clear inputs and outputs — no reliance on leftover notebook state.
  • Add type hints and a docstring so the function documents itself.
  • Follow PEP 8 (Python’s style guide) — a linter like ruff does this for you automatically.
  • Write a test so you know it still works after you change it.

That’s most of it. Let’s see it on a real example.

· Demo: the same logic, two ways

We’ll use a tiny piece of real work from later in the course: cleaning a “route” field from the city’s sites export. The raw values are messy — "2208", " 81 ", "abc12", floats like "2208.0". We want the last 4 digits, zero-padded. Run each version yourself as you read.

1 How it looks in a notebook

In a notebook you’d probably just write the logic inline in a cell and eyeball the result:

# notebook cell — exploratory, runs top to bottom
import pandas as pd

s = pd.Series(["2208", " 81 ", "abc12", "2208.0", ""])

# strip to digits, take last 4, pad to width 4
clean = (
    s.astype(str).str.strip()
     .str.replace(r"\.0$", "", regex=True)
     .str.replace(r"\D+", "", regex=True)
     .replace("", pd.NA)
     .str[-4:].str.zfill(4)
)
clean   # look at the output

This is fine for figuring out the logic. The problem: it only exists in this notebook, in this cell, next to a specific variable s. You can’t reuse it elsewhere without copy-pasting, and if you change it, nothing tells you whether you broke it.

2 The same logic as a package function

Now lift it into a named function in a module — this is essentially what clean_route4 in prep/sites.py is:

# prep/sites.py — reusable, documented, testable
import pandas as pd


def clean_route4(series: pd.Series) -> pd.Series:
    """Normalize a raw route field to its last 4 digits, zero-padded.

    Strips whitespace and non-digits; empty results become NA.
    """
    s = series.astype(str).str.strip()
    s = s.str.replace(r"\.0$", "", regex=True)   # drop trailing ".0" from floats
    s = s.str.replace(r"\D+", "", regex=True)    # keep digits only
    s = s.replace("", pd.NA)
    return s.str[-4:].str.zfill(4)

What changed, concretely:

  • It has a name and lives in a file — from opentrash.prep.sites import clean_route4 works from anywhere.
  • It takes an explicit input (series) and returns a value — no dependence on a notebook variable.
  • The type hint (series: pd.Series -> pd.Series) says what goes in and out.
  • The docstring explains it to the next person (often future you).

The actual computation is identical. You didn’t learn a new algorithm — you wrapped the one you had in a few conventions.

3 And a test, so it stays correct

The final habit: a small test pinning the behavior. Now any future change that breaks it gets caught instantly.

# tests/test_sites.py
from opentrash.prep.sites import clean_route4
import pandas as pd

def test_clean_route4():
    out = clean_route4(pd.Series(["2208", "abc12", "1234567"]))
    assert out.iloc[0] == "2208"
    assert out.iloc[1] == "0012"   # digits only, padded
    assert out.iloc[2] == "4567"   # last 4

Run it with pytest. Green means it works; red means you broke something. That feedback loop is the whole point of structuring code this way.

· A word on classes (just enough)

You’ll see classes here and there in the course. You don’t need to master object-oriented programming to follow along, but here’s the one-paragraph version so nothing feels foreign.

A function is a verb — it does something. A class is a noun — it bundles related data (and the operations on it) into one named thing. When you have a value with several parts that travel together, a class (or its lightweight cousin, a dataclass) is cleaner than juggling a loose dictionary.

· Example: a vehicle ID

Later, a vehicle ID like "815001" needs to be split into a fleet prefix ("815") and a number ("001"), and classified (automated vs manual). Instead of passing three loose variables around, we bundle them:

from dataclasses import dataclass

@dataclass(frozen=True)
class VehicleId:
    raw: str        # "815001"
    prefix: str     # "815"
    number: str     # "001"

v = VehicleId(raw="815001", prefix="815", number="001")
v.prefix     # "815"  — access fields by name

A @dataclass writes the boilerplate for you (constructor, equality, a readable repr). frozen=True makes it immutable — you can’t accidentally change it after creation. That’s genuinely all you need to know to read the code in this course; we build the real VehicleId in Lesson 4 and use it in the GPS lesson.

The same goes for a Protocol (an interface that says “any class with these methods counts”) — we’ll meet one when we make GPS data vendor-independent. We’ll explain it in context when it shows up. No need to front-load the theory.

· The takeaway

  • Notebook code and package code run the same logic — the difference is a few conventions: modules, clear inputs/outputs, type hints, docstrings, style, tests.
  • You don’t need deep OOP to follow this course. Functions carry most of the weight; classes and dataclasses show up only where they genuinely help.
  • The tooling (ruff for style, pytest for tests) does the tedious enforcement so you can focus on the logic.
This is the only lesson that’s purely about how we write code rather than what we’re building. From Lesson 1 on, every lesson adds real functionality to the package — and you’ll be applying these conventions without thinking about them.
