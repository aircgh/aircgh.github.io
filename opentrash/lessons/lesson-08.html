Lesson 8 of 12 · DuckDB Masterclass

Refined hits in, service patterns out — eight CTAS steps.

Given a refined-hits cache (parcels ↔ pings), find the service signature of every parcel: which vehicle, which day-of-week, which hour, weekly or biweekly. The detection is a tour of how to do real analytics with DuckDB — eight small CREATE TABLE AS SELECT steps, each one a clean transformation that builds on the previous. Set-based SQL, not pandas loops.

Time: ~120 min You'll touch: patterns/{config,detector,validator} Result: per-parcel weekly1/weekly2/biweekly

· Objective

Build three modules that together turn the refined-hits cache from Lesson 7 into a parcel-level pattern report.

  • patterns/config.py — a frozen PatternConfig dataclass bundling all knobs (stop bin, weekly thresholds, biweekly thresholds, scoring weights). Default values match the city’s working dataset.
  • patterns/detector.py — the CTAS pipeline. Eight named steps plus a thin file-in/file-out wrapper for scripts.
  • patterns/validator.py — a ValidationReport with coverage stats and biweekly diagnostics; mirrors cell 4 of the notebook.
The detector is intentionally a series of small, named functions — one per CTAS step — rather than one giant function. Each step is inspectable, testable, and replaceable. If you want to change how regularity is computed, you touch one 30-line function.

· Before you begin

  • You’ve completed Lesson 7 — you have a refined-hits parquet (real or synthetic).
  • No new dependencies. DuckDB is already core.
  • If you ran just the bootstrap Mode B from Lesson 7 once, you have everything you need to test this lesson against your own data. If not, the tests use synthetic refined hits with crafted weekly + biweekly patterns — you can develop the whole module without real data.

· Build it, step by step

1 Knobs in a dataclass — patterns/config.py

Every threshold and weight as a typed field. frozen=True so the config can’t drift mid-run. Three logical groups:

  • Stop binning — stop_bin_minutes=15. How tight to group consecutive pings into a single “visit”.
  • Weekly acceptance — weekly_min_regularity=0.25, weekly_min_weeks=6. Floor for what counts as a weekly pattern.
  • Biweekly acceptance + scoring — gap target/tolerance, min visits/weeks, phase-share floor, composite-score floor, weekly-cap, and the three scoring weights (default 0.45 / 0.35 / 0.20).

2 The pipeline — patterns/detector.py

The detector is a sequence of eight CTAS steps in DuckDB. Reading them in order is the masterclass:

refined
  -> stops              -- 15-min bins per parcel/vehicle
  -> daily_visits       -- one row per parcel/vehicle/day
  -> dow_stats          -- regularity per parcel/vehicle/DOW
  -> best_dow           -- the top DOW per parcel/vehicle
  -> best_days          -- daily_visits rows on that best DOW

Branch A (biweekly):
  -> biweekly_gaps      -- gap analysis between consecutive visits
  -> biweekly_phase     -- even/odd week share around anchor week
  -> biweekly_scored    -- composite score (filtered by thresholds)
  -> biweekly_best      -- top biweekly candidate per APN

Branch B (weekly):
  -> weekly_pool        -- best_dow rows with no biweekly winner
  -> weekly_ranked      -- ROW_NUMBER() per APN
  -> weekly1, weekly2   -- rank 1 and 2

Final:
  -> patterns           -- parcels LEFT JOIN weekly1 weekly2 biweekly

Each step is its own small Python function (_build_stops, _build_dow_stats, ...) that runs one CREATE TABLE AS SELECT. detect_patterns(con, refined_table, parcels_table, config) calls them in order.

3 Key SQL idioms to internalize

The CTAS pipeline uses a small kit of DuckDB / SQL features over and over. Worth recognizing:

  • Time-bucketing with epoch math: CAST(epoch(dt_local) / (bin * 60) AS BIGINT) AS bin_id — turns timestamps into a discrete bin index, zero allocations.
  • Window function ranking: ROW_NUMBER() OVER (PARTITION BY APN, VehicleName ORDER BY regularity DESC, ...) — the standard “top-N per group” pattern, used here for both best_dow and weekly1/2.
  • FILTER clauses on aggregates: MEDIAN(gap_days) FILTER (WHERE gap_days IS NOT NULL) — cleaner than nested CASE/COALESCE for conditional aggregation.
  • Phase math: date_diff('week', anchor, week_start) % 2 — classifies each visit week as even/odd around an anchor; GREATEST(even, odd) / total is the phase share.
  • LEFT JOIN for "outer presence": the final assembly uses parcels LEFT JOIN weekly1 LEFT JOIN weekly2 LEFT JOIN biweekly_best so every parcel appears in the output, with NULLs where no pattern was detected.

4 Batch-safe processing in DuckDB at scale

When you run this against real refined-hits data — tens of millions of rows over years — the same pattern from Lesson 7’s Mode B applies: don’t hold giant pandas frames in memory. Two techniques to lean on:

  • Stage as DuckDB tables, not Python frames. The detector creates CTAS tables on the connection; intermediate results live in DuckDB’s memory/spill model, not Python’s. Only the final patterns table comes back to pandas.
  • Configure DuckDB for the run. Before big jobs, set SET threads, SET memory_limit, and SET temp_directory on the connection so it can spill to disk safely. Then run the pipeline normally.
  • Idempotent re-runs. The detector calls DROP TABLE IF EXISTS before each CREATE, so re-running is safe. Combined with the chunked refined-hits cache from Lesson 7, an interrupted multi-step job picks up cleanly.

The 18-hour Mode B rebuild from Lesson 7 produces the refined-hits cache; pattern detection runs against that, in minutes for a year of data. The slow part runs once.

5 Diagnostics — patterns/validator.py

A small ValidationReport dataclass with everything in one place:

  • Counts and coverage % for weekly1, weekly2, biweekly, and all-three.
  • Weekly1 == Weekly2 sanity check — non-zero means the ranking has a bug; same vehicle shouldn’t hold both slots.
  • Optional describe() stats on biweekly score and phase share, dictified for clean logging.

The summary() method returns a printable multiline string for quick post-run inspection.

6 Test with crafted synthetic patterns

The detector tests synthesize three parcels with known patterns:

  • W: every Tuesday for 14 weeks — should detect as weekly1, regularity ≈ 1.0, no biweekly.
  • B: every other Tuesday for 14 weeks (7 visits) — should detect as biweekly, no weekly (excluded from the pool by the biweekly winner).
  • S: just two visits total — below all thresholds, no pattern detected (but the parcel still appears in the output via the LEFT JOIN).
