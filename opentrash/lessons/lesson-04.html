Lesson 4 of 12 · Excel → Parquet

Turn messy Excel drops into a durable, idempotent parquet store.

Tonnage data arrives as a steady stream of Excel files — one per day, from two systems (ARTS for external landfills, RAD for Miramar). In this lesson you’ll build a pipeline that ingests them into year-partitioned parquet, with stable hash keys that make re-running on the same files a no-op. The pattern — registry, cleaner per source, stable key, idempotent upsert — reappears everywhere good data engineering happens.

Time: ~75 min You'll touch: tonnage/{registry,cleaners,keys,upsert,pipeline} Result: idempotent annual parquet store

· Objective

Build the tonnage/ subpackage: a five-module pipeline that discovers Excel drops, cleans each by source, generates a stable per-record hash, and writes to year-partitioned parquet without ever creating a duplicate — even if you re-run on overlapping inputs.

  • registry — scan a folder, detect source from filename, parse the trailing date.
  • cleaners — per-source: ARTS keeps full route detail; RAD filters to our fleet by prefix and drops DEMO rows.
  • keys — sha1 hash over normalized fields (minute-floored time, tons rounded to 3, etc.).
  • upsert — year-partitioned parquet with set-difference on record_key.
  • pipeline — run_ingest() orchestrates the full run.
The theme is idempotency: re-running the pipeline on the same files (or files that overlap a prior run) must produce the same result without growing the store. Stable hash keys are the trick that makes it work.

· Before you begin

  • You’ve completed Lesson 3 — sites cleaning and the rules-as-data pattern are familiar.
  • Your environment is active.
  • You have at least one ARTS or RAD Excel file to work with. The tests run end-to-end against synthetic .xlsx files in a temp folder, so you can build and verify the whole subpackage with no real data on hand.

· Build it, step by step

1 Add the new dependencies

Reading Excel and writing parquet:

dependencies = [
    # ... existing ...
    "pyarrow>=15",       # parquet I/O
    "openpyxl>=3.1",     # reading .xlsx
]
pip install -e ".[dev]"

2 Write tonnage/registry.py

A scanner that turns a folder of files into a DataFrame describing each one. Detect source from the filename prefix (ARTS_, RAD_); parse the trailing MMDDYY into a file_date; capture modified_time too. The registry is the input every later step iterates over.

Filename convention is the source of truth here. Operations name files deliberately; we lean into that.

3 Write tonnage/cleaners.py

Two cleaners, one per source, sharing a small set of helpers:

  • _canon_header + resolve_column — case- and whitespace-tolerant header matching. Excel exports drift; rename one column upstream and a brittle reader breaks. We don’t.
  • clean_fleet_id, parse_dt, to_float — tiny, focused field cleaners.
  • matl_norm_arts / matl_norm_rad — normalize material descriptions into a 4-class label: REFUSE, RECYCLING, ORGANIC, OTHER.
  • clean_arts_file — load, keep known columns, parse dates, normalize material, attach lineage.
  • clean_rad_file — same shape, plus a fleet-prefix filter (RAD includes other haulers; we keep only our trucks) and a DEMO drop.

Each cleaner returns (df, qa_dict) — cleaned frame plus a small audit dict with row counts, null percentages, and which columns it actually used. Great for the registry log and great for debugging.

4 Write tonnage/keys.py

Every cleaned row gets a record_key — a sha1 over a small set of normalized fields. The two source-specific helpers add_keys_arts and add_keys_rad also add event_dt (minute-floored) and year columns for partitioning.

# ARTS key fields
event_dt_minute || fleet || route || load# || dump || material || tons(.3)

# RAD key fields
event_dt_minute || fleet || landfill || material || tons(.3) || tran#

Why minute-flooring and 3-decimal rounding? Two records of the same weighbridge event can differ by milliseconds or a trailing float digit. Without normalization, they’d hash differently and pollute the store with duplicates.

5 Write tonnage/upsert.py

The idempotent writer. For each year in a batch:

  1. If <SOURCE>_<year>.parquet exists, read just its record_key column.
  2. Set-diff the incoming batch against existing keys.
  3. Concatenate the existing rows with the new ones, dedupe by record_key for safety, rewrite the file.
  4. If the file doesn’t exist yet, just write the batch.
summary = upsert_annual_parquet(df, "ARTS", out_dir)
# {2026: {"existing": 6076, "added": 303, "total": 6379}}

A year of weighbridge events fits in memory comfortably, so the simple read-merge-rewrite approach is fine. It also guarantees the result is internally consistent even if a batch contains in-source duplicates.

6 Write tonnage/pipeline.py

run_ingest(data_dir, out_dir) ties it together: scan → clean each by source → add keys → in-batch dedupe → upsert per year → write the registry as registry.parquet alongside the year files. Per-file errors are caught and recorded in the registry (status = FAIL) without aborting the whole run.

7 Test against synthetic Excel files

Add tests/test_tonnage.py. Cover filename parsing, the canonical-header resolver, key stability (identical inputs → identical keys; trailing-digit jitter does not change keys), upsert idempotency, and a full end-to-end run that writes a few synthetic .xlsx files to a temp folder and runs the pipeline twice — the second run must add zero new rows.

pytest -v
git add .
git commit -m "Lesson 4: tonnage ingest pipeline with idempotent upsert"
git push origin main

· What you built

  • A registry that discovers and describes tonnage drops by filename.
  • Source-specific cleaners tolerant of header drift, each producing the same canonical shape.
  • Stable hash keys that survive trivial input variation — the key to idempotency.
  • A year-partitioned parquet store with idempotent upsert; safe to re-run on overlapping inputs.
  • A full pipeline with per-file error capture and an audit registry written alongside the data.
You now have a durable tonnage store that grows as new drops arrive. Lesson 5 turns the lessons-learned-so-far into real Python packaging: turning ad-hoc notebook code into proper modules, dataclasses, and protocols — the bridge between “works in Jupyter” and “works as a library.”
