Lesson 3 of 12 · Sites Data

Clean messy input, and turn business rules into code.

Real operational data is messy. In this lesson you’ll take the raw sites export — hundreds of thousands of rows, mixed quality — and turn it into a clean, validated table. Along the way you’ll learn the project’s most important pattern: encoding institutional business rules as data, so the knowledge in someone’s head becomes something the computer enforces.

Time: ~60 min You'll touch: prep/sites · tests Result: clean sites + valid route IDs

· Objective

Build prep/sites.py: a pipeline that turns the raw sites CSV into a clean master table with canonical route IDs for refuse, recycling, and organics — each validated against the operation’s real rules.

  • Select and rename the columns that matter; drop the rest.
  • Clean the route, day, and week fields into normalized forms.
  • Assemble route IDs by concatenation, then validate them against business rules.
  • Filter out coordinates that fall outside the service region.
The theme is rules as data. Instead of scattering if statements through the code, the valid supervisor prefixes, sequence thresholds, and day/week codes live in named sets and constants. Anyone can read them, audit them, and adjust them as operations change — without touching the logic.

· Before you begin

  • You’ve completed Lesson 2 — the spatial foundation is in place.
  • Your environment is active: conda activate opentrash.
  • You have a sites export (CSV) to work with. The tests use synthetic rows, so you can build and verify the whole module without real data; point it at your real export when you’re ready.

· Build it, step by step

1 Understand the route ID

Before writing code, understand what you’re parsing. A route ID like 22081 or 22433O packs several facts into one code:

  • Digits 1–2 — supervisor & commodity zone. Ranges 21–25 and 11–12 handle refuse and recycling; 31–35 handle organics only. 11/12 are the manual collection routes.
  • Digits 3–4 — sequence number, which also disambiguates commodity: < 40 = refuse, ≥ 40 = recycling. For manual routes, < 40 refuse, 40–59 recycling, ≥ 60 organics.
  • Digit 5 — collection day, 1–5 (Mon–Fri).
  • Trailing letter — biweekly week, O (orange) or B (blue). Recycling only, because it alternates every other week.

This is the institutional knowledge the lesson captures. Writing it down as code is what makes it durable — it no longer lives only in an operator’s memory.

2 Define the column map and the rules as data

At the top of sites.py, declare a COLUMN_MAP (raw name → clean name) and the business rules as named constants — the valid prefix sets per commodity, the manual-route prefixes, the day/week value sets, and the regional bounding box.

REF_PREFIXES = {"21","22","23","24","25","11","12"}
ORG_PREFIXES = {"31","32","33","34","35","11","12"}
MANUAL_PREFIXES = {"11","12"}
VALID_DAYS = {"1","2","3","4","5"}
VALID_WEEKS = {"B","O"}

3 Write the field cleaners

Small, focused functions that normalize one thing each:

  • clean_route4 — strip to digits, take the last 4, zero-pad.
  • clean_day — a single digit 1–5, else NA.
  • clean_week — uppercase B or O, else NA.

Each cleaner returns NA for anything it can’t make sense of, rather than guessing. Bad input becomes missing data you can count — not silent corruption.

4 Assemble the route IDs

Write build_route_ids: refuse and organics are route4 + day; recycling is route4 + day + week (the extra letter for the biweekly cycle). A row only gets an ID for a commodity when all its parts are present.

5 Write the validators

Three functions — valid_ref, valid_rec, valid_org — each returning a boolean mask over a route-ID series, applying the prefix + sequence + day (+ week) rules. The manual-route (11/12) sequence bands are the subtle part: refuse < 40, recycling 40–59, organics ≥ 60.

Returning masks (not filtered frames) keeps the validators composable — the caller decides what to do with invalid rows: drop them, log them, or report the offenders.

6 Tie it together and filter junk coordinates

Write clean_sites(df_raw) to run the whole pipeline: select + rename, clean the parts, parse coordinates, build IDs, and drop rows whose x/y fall outside the regional bounding box (this catches (0,0) placeholders and typos). Add load_sites(path) to read a CSV and return the cleaned master.

7 Test, then commit

Add tests/test_sites.py with synthetic rows covering each cleaner, route-ID assembly, every validator (including the manual-route edge cases), and the bbox filter. Then:

ruff check .
pytest -v
git add .
git commit -m "Lesson 3: sites cleaning + route-ID rules as data"
git push origin main
